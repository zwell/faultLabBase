# CLAUDE.md — FaultLab Web

本文件是 FaultLab Web 前端的 Claude Code 开发规范。
**每次开始任务前必须完整读取。**

---

## 产品定位

面向工程师的故障排查训练平台。
用户打开页面的心态是「我要开始一个故障演练」，不是「我要学习一个工具」。

**核心设计原则：用户只需要关注故障本身，其他全部隐藏。**

---

## 用户流程（页面对应关系）

```
选底座（首页）
  → 选场景（场景列表页）
    → 读剧本，开始排查（场景详情页）
      → 提交分析，获得反馈（verify 面板，内嵌在详情页）
```

共三个页面，不要拆更多。

---

## 页面规范

### 页面一：首页（选底座）

**用途**：让用户选择要进入的业务环境。

**显示内容**：
- 每个底座一张卡片
- 卡片内容：底座名称、一句话描述、包含的场景数量、资源等级
- 底座状态：运行中 / 未启动（从后端轮询）

**不显示**：
- 技术栈列表
- 版本号
- 任何配置说明

**交互**：
- 点击卡片 → 进入场景列表页
- 若底座未启动，点击后显示启动进度，完成后自动跳转

**卡片示例**：
```
电商系统
模拟大促场景下的订单、支付、库存链路
12 个场景  ·  重量级
[开始演练]
```

---

### 页面二：场景列表页

**用途**：在当前底座下选择一个故障场景。

**显示内容**：
- 场景卡片列表
- 每张卡片：故障现象（业务语言）、难度、预计时长
- 筛选：难度（L1–L5）、业务链路（下单 / 支付 / 搜索 / 库存）

**不显示**：
- 技术栈标签（用户不应该从技术栈角度选场景）
- 资源等级（底座已经选了，不用重复显示）
- 场景 ID

**卡片文案用业务语言**：
```
✗  Kafka 消费者 Rebalance 风暴        ← 技术语言，不用
✓  积分、物流消息延迟数小时            ← 业务现象，用这个
```

**卡片内容**：
```
积分、物流消息延迟数小时
支付链路  ·  L3 中等  ·  30–45 分钟
```

---

### 页面三：场景详情页（核心页面）

**布局**：左右分栏，比例 45% / 55%。

#### 左栏：剧本 + 排查线索

顶部显示「业务剧本」（scenario.md 内容），用醒目的方式呈现，
让用户感受到这是一个真实的 on-call 情境。

剧本之后是「排查路径」：可复制的命令列表，无解读，无提示。

**不显示**：
- 技术栈、难度、资源等级等元信息标签（进来就排查，不需要这些）
- 「故障现象」标题（剧本本身就是现象）
- 任何根因线索

#### 右栏：终端 + 问题分析

上半部分：终端（连接容器，可直接执行命令）。

下半部分：分析输入框 + 发送按钮。
- 占位符文字：「描述你的判断…」
- 不要写「输入分析...（Ctrl+Enter 发送）」这类说明性文字，直接简洁

**终端区域**：
- 容器切换用 Tab 或下拉，不要并排显示多个终端
- 顶部只保留「重启」「清理」两个操作按钮，去掉所有其他控件
- 连接成功后不显示 FaultLab 的欢迎文字，直接进入 shell

#### verify 反馈（内嵌，不跳新页面）

用户发送分析后，右栏下半部分变为对话模式：
- LLM 回复逐字流式显示
- 回复风格：mentor，不是考官——不打分，只引导
- 若用户方向错了，给下一步提示，不直接说答案
- 若用户答对了，给出完整的根因解释 + 推荐阅读

## 视觉规范

**延续现有风格**：
- 深色背景（#0d1117 或接近）
- 卡片用略亮的背景色区分层次
- 主色调蓝紫色（现有按钮色）
- 字体：系统默认等宽字体用于代码，无衬线字体用于正文

**减少视觉噪音**：
- 去掉所有装饰性图标（技术栈 logo、资源等级 emoji 等）
- 难度：列表与详情用 **L1–L5 色标胶囊**（绿→蓝→黄→橙→红），同时保留 **Lv + 文字**（如「L3 中等」），并带 `title` 提示。色标用于快速扫视，不以颜色为唯一信息渠道
- 进度条 / 难度条用实心点表示（现有设计已有，保留）

**信息密度**：
- 卡片上的信息控制在 3 行以内
- 详情页左栏不超过 2 个 H2 标题（剧本、排查路径）

## 技术约束

**框架**：与现有项目保持一致（Vue / React / 原生，按现有项目来）。

**后端 API 约定**（前端视角）：

```
GET  /api/basecamps                    → 底座列表 + 运行状态
POST /api/basecamps/:id/start          → 启动底座
GET  /api/basecamps/:id/scenarios      → 场景列表
POST /api/scenarios/:id/inject         → 注入故障
POST /api/scenarios/:id/clean          → 重置场景
GET  /api/scenarios/:id/terminal       → WebSocket，终端连接
POST /api/basecamps/:id/scenarios/:scenarioId/verify → 提交分析，SSE 流式（`text/event-stream`；`token` / `done` / `error`）
```

**终端**：使用 xterm.js，连接后端 WebSocket。

**verify 流式输出**：使用 SSE（Server-Sent Events），逐 token 渲染。

## 禁止事项

- 不在用户界面上暴露 inject.sh、meta.yaml、SOLUTION.md 等文件名
- 不显示场景 ID（如 mysql-001）
- 不在剧本页面显示「技术栈：MySQL」「资源等级：重量」等标签
- 不在卡片上显示超过 3 行信息
- 不做引导弹窗、新手教程、tooltip 说明
- 不在终端连接成功后打印欢迎语
- verify 反馈不打百分制分数，不说「你答对了 X 分」
- 不在页面上放置「关于 FaultLab」「使用文档」等链接入口

---

## 文案规范

- 用业务语言描述故障，不用技术术语作为标题
- 按钮文字：动词开头，简短（「开始演练」「重置」「发送」）
- 空状态文字：说清楚用户下一步做什么（「选择一个场景开始」）
- 错误提示：说人话，不暴露技术细节（「底座启动失败，请检查 Docker 是否运行」）
